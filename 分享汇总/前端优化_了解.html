前端性能优化

1. 了解

  （1）前端性能优化的涉及点 "服务器" 到 "协议" 再到 "宿主环境本身" 都要有比较深刻的认识，业界目前主要还是以雅虎总结出来35条前端性能优化黄金条例。
  （2）web性能优化有个大家比较公认的二八原则，就是资源从服务器处理完下发到客户端的浏览器上所占的时间比例大概是整个过程的20%，也就是说服务器端可以优化的空间的效率提升并不会很明显，前端性能优化成为web性能优化重点考虑的领域。

2. 浏览器渲染与性能优化

关键渲染路径 ( CRP ) - Critical Rendering Path

  1. 概述： 浏览器通过HTTP协议发送页面请求，接收到服务器返回的HTML、CSS和JavaScript字节据并对其进行解析和转变成像素的渲染过程被称为关键渲染路径。通过优化关键渲染路径即可以缩短浏览器渲染页面的时间
    （1） 构建文档对象模型 (DOM)
    （2） 构建 CSS 对象模型 (CSSOM)
    （3） 构建渲染树 (Render Tree)
    （4） 布局 (layout)
    （5） 绘制 (painting)

  2. DOM树 - Document Object Model（文档对象模型）. 浏览器完成页面的所有后续处理都是建立在这个DOM基础上的。

  3. CSSOM树 - Cascading Style Sheets Object Model(层叠样式表对象模型)，它是CSS的对象集合。
    浏览器获得外部CSS文件的数据后，浏览器预测会需要这个资源来渲染页面，因此会立即发出一个该资源的请求，就会像构建DOM树一样开始构建CSSOM树。

  1. 构建渲染树 （Render Tree ）
    想要渲染出页面，就需要将DOM树与CSSOM树结合在一起，即 Render Tree。
    （1）从 DOM Tree 的根节点开始，遍历每个可见的节点，为每一个可见的节点匹配并应用对应的 CSSOM 规则。
    （2）某些不可见的节点如 script 标签、meta 标签等和某些通过 CSS 隐藏的节点如应用了 display:none 规则的节点不会体现在渲染结果中，因此会被忽略。

  2. 布局 （layout）
    渲染树完成之后 , 我们还要计算它们在视口（viewport）中的准确位置及尺寸大小——这就是布局（layout）阶段要做的工作，也就是常说的重排（reflow）。

    布局阶段会从渲染树的根节点开始遍历，然后确定每个节点对象在页面上的确切大小与位置
    布局阶段的输出是一个盒子模型，它会精确地捕获每个元素在屏幕内的确切位置与大小，所有相对的测量值也都会被转换为屏幕内的绝对像素值。

  3. 绘制 （painting）
    当Layout布局事件完成后，浏览器会立即发出 Paint 事件，开始将渲染树绘制成像素，绘制所需的时间跟CSS样式的复杂度成正比，绘制完成后，用户就可以看到页面的最终呈现效果了。

  4. 回流(reflow)：当页面中某些元素发生变化而影响了布局时(如尺寸、位置改变)，就会发生回流，浏览器需要重新布局并绘制。

  5. 重绘(repaint)：当页面中某些元素发生了不影响布局的变化时(如颜色改变)，就会发生重绘，浏览器需要重新绘制。

优化CRP(页面渲染速度)
1. HTML与CSS文件结构非常庞大与复杂，这显然会给页面加载速度带来严重影响。
2. HTML、CSS、JavaScript都是会对渲染产生阻塞的资源，HTML是必需的（没有DOM还谈何渲染），但还可以从CSS与JavaScript着手优化，尽可能地减少阻塞的产生。
3. JS部分:
当浏览器的HTML解析器遇到一个script标记时会暂停构建DOM，然后将控制权移交至JavaScript引擎，这时引擎会开始执行JavaScript脚本，直到执行结束后，浏览器才会从之前中断的地方恢复，然后继续构建DOM。
每次去执行JavaScript脚本都会严重地阻塞DOM树的构建，如果JavaScript脚本还操作了CSSOM，而正好这个CSSOM还没有下载和构建，浏览器甚至会延迟脚本执行和构建DOM，直至完成其CSSOM的下载和构建。显而易见，如果对JavaScript的执行位置运用不当，这将会严重影响渲染的速度。
优化方案：
（1） CSS放在顶部：优先渲染 link 标签
（2） JS放在底部： 避免阻塞
（3） img标签要设置高宽：减少重绘重排
（4） 减少DOM元素和DOM操作：这个最能体现编程水平了
前端框架 vue、react，虚拟dom的渲染方案，在内存中进行 dom diff 比较，做到最小化的操作真实的 dom （操作真实的 dom 常常会成为性能瓶颈），能极大的提高渲染速度
（5）利用事件冒泡 ： 利用事件冒泡 在目标DOM元素的父标签（大部分框架是在body标签统一注册事件监听）注册回调，收拢事件监听入口同时节约了DOM节点引用开销。
静态资源优化
主要是减少静态资源的加载时间，主要包括html、css、js和图片文件，静态资源的加载时间是前端性能最大的瓶颈（特别是图片），现如今优化的手段也很丰富，以下简要列举几种常用的方法

（1）减少 HTTP请求数 : 合并css、js文件，制作雪碧图：减少http的请求次数，节省网络请求时间
80%的响应时间花在下载网页内容(images, stylesheets, javascripts, scripts, flash等)。减少请求次数是缩短响应时间的关键！可以通过简化页面设计来减少请求次数，但页面内容较多可以采用以下技巧。

  • 1、 CSS Sprites 雪碧图
    就是把多个图片拼成一副图片，然后通过CSS来控制在什么地方具体显示这整张图片的什么位置
  • 2、 CSS、JS文件合并
    现在有很多现成的库可以帮你将多个脚本文件捆绑成一个文件，将多个样式表文件捆绑成一个文件，以此来减少文件的下载次数。（webpack / gulp）
  • 3、font-icon字体图标
    体验好，可以使用CSS的字体样式， 淘宝自身的许多小图标都是用的字体图标来展示的。
  • 4、图片 base64编码传输
    将图片数据 编码成一串字符串，使用该字符串替代图片地址。
    意义就是 减少一次http 请求。（图片下载就要发请求）
    何时使用
  1. 这类图片不能与其他图片以CSS Sprite的形式存在，只能独行
  2. 这类图片从诞生之日起，基本上很少被更新
  3. 这类图片的实际尺寸很小
  4. 这类图片在网站中大规模使用
  5. 对于极小或者极简单的图片，例如只有几像素的图片，不用考虑跨域问题也不想页面的图片缓存，果断用base64。
  • 5、图片延时加载 （懒加载）
    主要是为了减少首屏一次性图片的加载量，按需加载图片
